# 🛑 지옥의 자가 충돌(Self-Conflict) 방지 프로토콜

혼자 작업하거나 소규모 팀에서 브랜치가 꼬여 지옥 같은 충돌(Conflict)이 발생하는 것을 막기 위한 **절대 방어 프로토콜**입니다. 새로운 작업을 시작할 때 이 단계를 순서대로 체크하며 진행하세요.

---

## 단계 1: 작업 시작 전 (Pre-flight Check)
> **목적:** 내 로컬 환경을 "가장 깨끗한 최신 상태"로 리셋하기

1. [ ] 하던 작업을 멈추고 `develop` (또는 `main`) 브랜치로 이동합니다.
   ```bash
   git checkout develop
   ```
2. [ ] 원격 저장소의 다른 사람(또는 과거의 나)이 올려둔 최신 내용을 모두 로컬로 끌어옵니다.
   ```bash
   git pull origin develop
   ```
3. [ ] (선택) 더 이상 안 쓰는 과거 브랜치는 혼동을 방지하기 위해 로컬에서 삭제해 둡니다.

---

## 단계 2: 브랜치 생성 (Branch Out)
> **목적:** 최신 지점을 베이스로 삼고, 오직 한 가지 작업만 할 공간 만들기

1. [ ] 최신화된 베이스 브랜치에서 새 작업 브랜치를 무조건 파생시킵니다.
   ```bash
   git checkout -b feat/task-name
   # 예: git checkout -b feat/audit-log-email
   ```
2. [ ] **절대 금지:** 어제 작업하던 낡은 `feat/A` 위에서 그대로 `feat/B`를 시작하지 마세요.

---

## 단계 3: 작업 중 규율 (In Progress Rules)
> **목적:** "나중에 한 번에 터질 수류탄"을 심지 않기

1. [ ] **관심사 분리 (단일 책임 원칙):** 현재 브랜치의 목적과 관계없는 폴더나 파일을 수정하지 않습니다. 
2. [ ] **공통 파일 보호:** `types.ts`, `constants.ts` 등 공통 파일을 수정하고 싶은 유혹이 든다면, 멈추세요!
   - 🚨 **해결책:** 기존 작업 중인 코드를 안전하게 임시 저장(`git commit` 또는 `git stash`)하고, `develop`으로 돌아가 **다같이 쓸 공통 파일만 수정하는 새 브랜치**를 우선적으로 따세요. 그 브랜치를 먼저 머지시킨 뒤, 원래 하던 작업 브랜치로 돌아와 최신 공통 코드를 받아야 합니다.
3. [ ] `feat/A`에서 필요한 코드가 있다고 임의로 `feat/B`를 내 브랜치에 직접 머지 시키지 마세요. 족보가 꼬입니다.

---

## 단계 4: 매일 아침 동기화 (Daily Long-running Sync)
> **목적:** 내 브랜치가 혼자 과거에 머물러 고립되는 것 방지

이틀(2일) 이상 걸리는 작업이라면, 매일 아침 작업 시작 전 다음을 실행합니다. 거대한 폭탄을 쪼개서 매일매일 작은 폭죽으로 맞기 위한 전략입니다.

1. [ ] 변경 사항을 가져오고 현 브랜치에 `develop` 최신화 반영하기
   ```bash
   git fetch origin
   git merge origin/develop
   ```
   이때 자잘한 충돌이 나면 그 즉시 해결해서 다시 깔끔한 백지를 유지하세요.

---

## 단계 5: 커밋 전 점검 (Pre-Commit Check)
> **목적:** 우발적으로 튕긴 돌(잘못된 전체 포매팅, 더미 파일)이 들어가는 것 걸러내기

1. [ ] 터미널이나 에디터 소스 트리를 열어 변경된 리스트(`git status`)를 눈으로 스캔합니다.
2. [ ] **"내가 만진 기억이 없는데?"** 싶은 파일이 한 개라도 있다면 멈추세요. (`git diff <file>`)
3. [ ] 에디터의 자동 정렬 기능 때문에 딱 한 마디 수정했는데 파일 전체 줄바꿈이 바귀었다면, 변경 사항을 되돌리고(Discard / Restore) 원상복구 시킨 뒤 진짜 필요한 부분만 커밋합니다.

---

## 단계 6: PR 전송 후 점검 (Pre-Merge Sync)
> **목적:** 최종 방어막 넘기기

1. [ ] 깃허브 저장소에 PR을 올린 뒤, 봇이 띄워주는 **"This branch has conflicts that must be resolved"** 경고 문구가 뜨는지 확인합니다.
2. [ ] 만약 충돌이 났다면 브라우저상에서 에디터를 켜서 수정하지 말고, 터미널로 돌아와 안전하게 `develop` 기준을 한 번 더 당겨(`merge origin/develop`) 로컬에서 테스트한 후 머지합니다.
