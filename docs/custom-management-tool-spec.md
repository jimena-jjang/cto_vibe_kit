## 1. 프로젝트 개요

### 1.1 배경 및 목적

bemyfriends(이하 BMF)는 **b.stage** 플랫폼 위에서 외부 파트너/고객사가 커스텀 페이지를 개발·배포할 수 있는 환경을 제공하고 있다. 현재 이 과정에서 발생하는 **라우팅 매핑, S3/CDN 설정, GitHub 레포지토리 관리, 배포 파이프라인 설정** 등의 작업이 DevOps 팀의 수기 작업으로 진행되고 있어 다음과 같은 문제가 발생한다.

| 문제 영역 | AS-IS 현황 |
|-----------|-----------|
| **라우팅 매핑** | 내부 개발자가 설정 파일/코드를 직접 수정하여 매핑 |
| **레포지토리 관리** | GitHub에 코드를 fork한 후 수동으로 진행 |
| **배포** | GitHub Actions + Runner → S3 배포, 수동 CDN 설정 |
| **버전 관리** | 파일 단위의 비체계적 관리 |
| **긴급 대응** | Kill Switch 등 긴급 제어 수단 부재 |

**내부 개발 포털(Inhouse Dev Portal)** 은 이러한 수기 작업을 **도구(UI + API)** 로 전환하여 업무 가시성과 운영 안정성을 확보하는 것을 목표로 한다. 그 첫 번째 모듈이 **커스텀 관리도구**이다.

### 1.2 프로젝트 범위

```
┌─────────────────────────────────────────────────────┐
│                  내부 개발 포털                         │
│  ┌───────────────────────────────────────────────┐   │
│  │         ★ 커스텀 관리도구 (1차 구현 대상)        │   │
│  │  • 커스텀 프로젝트 생성/관리                     │   │
│  │  • GitHub Repo 자동 생성/권한 관리               │   │
│  │  • CI/CD 파이프라인 관리                         │   │
│  │  • S3/CDN 리소스 관리 및 배포                    │   │
│  │  • Route URI ↔ 프로젝트 매핑                    │   │
│  │  • Kill Switch / 버전 관리                       │   │
│  └───────────────────────────────────────────────┘   │
│  ┌─────────────────┐  ┌──────────────────────────┐   │
│  │  추후 모듈 A     │  │   추후 모듈 B              │   │
│  │  (예: 모니터링)   │  │   (예: 인프라 관리)         │   │
│  └─────────────────┘  └──────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

### 1.3 핵심 목표

1. **커스텀 프로젝트의 라이프사이클 관리 자동화** – 프로젝트 생성부터 배포, 운영까지 한 곳에서 관리
2. **세 가지 커스텀 개발 타입 지원** – Full Page, In-page, API 연동
3. **DevOps 반복 업무 도구화** – CI/CD, S3/CDN, 라우팅 매핑의 GUI 기반 관리
4. **GitHub 연동 자동화** – 레포지토리 생성, 권한 부여, 배포 파이프라인 자동 구성

## 2. 커스텀 프로젝트 타입 정의

b.stage에 외부 개발 결과물을 적용하는 **세 가지 방식**을 프로젝트 타입으로 정의한다.

### 2.1 Full Page (풀 페이지)

전체 페이지 단위로 커스텀 페이지를 퍼블리싱하는 방식.

| 항목 | 내용 |
|------|------|
| **설명** | 전체 페이지를 외부에서 개발하여 b.stage의 특정 경로에 매핑 |
| **템플릿 유형** | **a. Liquid 형식** – `template.liquid` + `data.json` 조합 (개발 완료, 3월 타겟) |
|  | **b. SDK(Web Component) 형식** – `manifest.json` + JS 번들 패키지 (개발 완료, 보안 진행 중) |
| **배포 산출물** | Liquid: 단일 파일 / SDK: 패키지(vendor/templateName, version) 단위 |
| **라우팅** | `repository : URI = 1:1` (추후 1:N 고도화) |
| **사용 시나리오** | 프로모션 페이지, 이벤트 페이지, 커스텀 랜딩 페이지 등 |

### 2.2 In-page (인페이지)

기존 b.stage 페이지의 특정 영역(Placeholder)에 컴포넌트를 삽입하는 방식.

| 항목 | 내용 |
|------|------|
| **설명** | 기존 페이지 내 지정된 영역에 외부 개발 컴포넌트 삽입 |
| **개발 상태** | 진행 중 |
| **배포 산출물** | Web Component 패키지 |
| **사용 시나리오** | 기존 페이지에 커스텀 위젯, 배너, 인터랙티브 요소 추가 |

### 2.3 API 연동

b.stage Gateway를 통해 외부에서 토큰 기반으로 API를 호출하여 데이터를 연동하는 방식.

| 항목 | 내용 |
|------|------|
| **설명** | b.stage Gateway를 통한 외부 API 호출 연동 |
| **개발 상태** | 진행 중 |
| **인증 방식** | 토큰 기반 (API Key / OAuth) |
| **사용 시나리오** | 외부 서비스에서 b.stage 데이터 활용, 커스텀 백엔드 연동 |

## 3. 기능 상세 명세

### 3.1 커스텀 프로젝트 관리

#### 3.1.1 프로젝트 생성

```
[프로젝트 생성 플로우]

사용자 → "새 프로젝트 생성" 클릭
  → 프로젝트 타입 선택 (Full Page / In-page / API)
    → Full Page 선택 시: 템플릿 유형 선택 (Liquid / SDK)
  → 프로젝트 기본 정보 입력
    - 프로젝트 이름
    - 설명
    - 대상 b.stage 채널
    - 환경 선택 (DEV / QA / REAL)
  → GitHub 레포지토리 자동 생성
    - 레포 이름: 시스템 자동 발번
    - 템플릿 기반 초기 코드 구조 생성
  → CI/CD 파이프라인 자동 구성
  → 프로젝트 대시보드로 이동
```

#### 3.1.2 프로젝트 목록/대시보드

| 기능 | 설명 |
|------|------|
| **프로젝트 목록** | 전체 프로젝트 리스트 (필터: 타입, 상태, 채널) |
| **프로젝트 상태** | Active / Inactive / Deploying / Error |
| **Quick Actions** | On/Off 토글, 배포, 버전 롤백 |
| **검색/정렬** | 이름, 타입, 생성일, 최근 배포일 기준 |

#### 3.1.3 프로젝트 상세

| 탭 | 기능 |
|----|------|
| **개요** | 프로젝트 기본 정보, 상태, 최근 활동 |
| **배포** | 배포 이력, 현재 활성 버전, 롤백 |
| **라우팅** | URI 매핑 설정, 진입점 설정 |

### 3.2 GitHub 레포지토리 관리

프로젝트 생성 시 GitHub 레포지토리를 자동으로 생성하고, 외부 파트너에게 권한을 부여하는 기능.

#### 3.2.1 자동 레포지토리 생성

```
[AS-IS]
DevOps 수기 → GitHub fork → 수동 설정 → 파트너 초대

[TO-BE]
어드민에서 프로젝트 생성
  → GitHub API를 통한 레포 자동 생성 (partners-bmf org 하위)
  → 레포 이름 시스템 자동 발번 (규칙: {project-type}-{channel}-{seq})
  → 프로젝트 타입별 템플릿 기반 초기 구조 생성
  → GitHub Actions workflow 파일 자동 세팅
  → 외부 파트너 초대 (Collaborator 권한)
```

#### 3.2.2 권한 관리

| 역할 | 권한 |
|------|------|
| **BMF 관리자** | Full access, 설정 변경, Kill Switch |
| **외부 파트너** | 코드 Push, PR 생성 (Write 권한) |
| **DevOps** | 강제 차단, 강제 롤백, 설정 Lock |

#### 3.2.3 GitHub 연동 기능 목록

- [ ] 레포지토리 자동 생성 (Organization 하위)
- [ ] 프로젝트 타입별 boilerplate 코드 자동 초기화
- [ ] 외부 파트너 Collaborator 초대/제거
- [ ] GitHub Actions workflow 자동 구성
- [ ] Webhook 연동 (Push/PR/Tag 이벤트 수신)
- [ ] Branch protection rules 자동 설정

### 3.3 CI/CD 파이프라인 관리

기존 3rd-Party CI/CD 플로우(`GitHub → Runner → S3/CDN`)를 관리도구에서 가시화하고 제어한다.

#### 3.3.1 파이프라인 개요

```
[CI/CD 파이프라인 흐름]

외부 파트너가 코드 Push
  → GitHub Actions 트리거
    → Runner에서 빌드 실행
      → 빌드 결과물을 S3에 업로드
        → CDN 배포 (캐시 무효화 포함)
          → 관리도구에 상태 콜백
```

#### 3.3.2 관리도구에서 제공하는 CI/CD 기능

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| **빌드 상태 모니터링** | 실시간 빌드 진행 상태 표시 | P0 |
| **빌드 이력 조회** | 과거 빌드 기록 및 로그 조회 | P0 |
| **배포 트리거** | 특정 태그/버전의 수동 배포 트리거 | P1 |
| **배포 롤백** | 이전 안정 버전으로 원클릭 롤백 | P1 |
| **파이프라인 설정** | 빌드 스크립트, 환경 변수 등 설정 관리 | P2 |

#### 3.3.3 버전 관리

- Git 태그(Tag) 기반 버전 관리 수행
- 라우팅 맵에서 "현재 이 경로는 어느 레포의 어느 태그(버전)를 바라본다" 형태로 참조 정보 관리
- 버전 히스토리 조회 및 특정 버전으로의 롤백 지원

### 3.5 Route URI 매핑 관리

커스텀 프로젝트의 결과물을 b.stage의 실제 서비스 경로에 연결하는 핵심 기능.

#### 3.5.1 매핑 구조

```
[라우팅 매핑 모델]

Route Mapping = {
  URI Path      : "/2026-promo"
  Project       : "promo-event-001"
  Project Type  : "Full Page (Liquid)"
  Active Version: "v1.2.0" (Git Tag)
  Status        : Active / Inactive
  Entry Point   : "메인 덮어쓰기" / "GNB 메뉴 추가"
  Environment   : REAL
  Device Filter : All / Desktop / Mobile
  Schedule      : { start: "2026-03-15 00:00 KST", end: "2026-04-15 23:59 KST" }
}
```

#### 3.5.2 매핑 기능 상세

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| **URI 매핑 생성** | 프로젝트와 URI 경로를 연결 | P0 |
| **매핑 On/Off** | 특정 매핑의 활성/비활성 토글 | P0 |
| **버전 선택** | 매핑에 연결할 프로젝트 버전(태그) 선택 | P0 |
| **진입점 설정** | 메인 덮어쓰기, GNB 메뉴 추가 등 진입 방식 설정 | P1 |
| **디바이스별 제어** | 디바이스별 GNB 노출 제어 | P1 |
| **타임존 예약** | 타임존 기반 On/Off 예약 | P2 |
| **프리뷰** | 실시간 미리보기 (Sandbox) | P2 |

#### 3.5.3 프로젝트 타입별 매핑 방식

| 프로젝트 타입 | 매핑 대상 | 동작 방식 |
|---------------|-----------|-----------|
| **Full Page (Liquid)** | URI Path → S3의 `template.liquid` + `data.json` | Gateway가 매핑 정보 참조 → S3에서 파일 로드 → 렌더링 |
| **Full Page (SDK)** | URI Path → S3의 JS 번들 패키지 | Gateway가 매핑 정보 참조 → CDN에서 패키지 로드 → Web Component 렌더링 |
| **In-page** | Placeholder ID → 컴포넌트 패키지 | 페이지 로드 시 Placeholder 감지 → CDN에서 컴포넌트 로드 → 삽입 |
| **API** | API Endpoint → Gateway 라우팅 규칙 | Gateway가 토큰 검증 → 외부 API 라우팅 |

### 3.6 Kill Switch 및 운영 제어

#### 3.6.1 Kill Switch

| 기능 | 설명 | 구현 레벨 |
|------|------|-----------|
| **프로젝트 단위 On/Off** | 특정 프로젝트를 즉시 비활성화 | 관리도구 UI |
| **매핑 단위 On/Off** | 특정 URI 매핑만 비활성화 | 관리도구 UI |
| **상위 레벨 차단** | WAF/라우터 레벨의 전체 차단 | DevOps + 관리도구 (TBD) |

#### 3.6.2 롤백 전략

- **MVP (3월 런칭)**: On/Off 토글 + 이전 버전 선택 배포
- **차기**: 원클릭 롤백 (이전 안정 버전으로 자동 전환)

#### 3.6.3 Audit Log

모든 설정 변경 이력을 기록:
- **Who**: 어떤 사용자가
- **When**: 언제
- **What**: 어떤 URL/패키지/설정을
- **How**: 어떻게 변경했는지

## 4. 권한 및 보안 모델

### 4.1 접근 제어

| 단계 | 방식 |
|------|------|
| **초기 (MVP)** | 회사 계정 SSO (Google Workspace) 기반 내부 전용 |
| **이후** | RBAC(역할 기반 접근 제어) 점진적 확장 |

### 4.2 권한 모델 (초기)

```
[권한 구조]

Admin (BMF 관리자)
  ├── 프로젝트 생성/수정/삭제
  ├── 라우팅 매핑 관리
  ├── GitHub 레포/권한 관리
  ├── Kill Switch 제어
  └── Audit Log 조회

DevOps
  ├── 모든 Admin 권한
  ├── 강제 차단 / 강제 롤백
  ├── 설정 Lock (변경 방지)
  ├── 환경별 인프라 설정
  └── 상위 레벨 Kill Switch

Viewer (향후 외부 파트너용)
  ├── 프로젝트 조회 (본인 소속만)

│  [5] 서비스 (자동)                                                │
│  └→ Gateway가 매핑 정보 참조 → S3/CDN에서 리소스 로드 → 렌더링       │
│                                │                                 │
│                                ▼                                 │
│  [6] 운영 (관리자)                                                │

│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 11. DevOps 협의 필요 사항 (체크리스트)

다음 항목은 DevOps 팀과 협의 후 확정이 필요한 사항이다.

- [ ] 커스텀의 범위와 DevOps 내 고려 범위 확정
- [ ] 3rd-Party CI/CD 파이프라인 재확인 (GitHub → Runner → S3/CDN 흐름)
- [ ] S3 버킷/CDN 도메인 경로 구조 및 네이밍 규칙 확정
- [ ] 환경별(DEV/QA/REAL) 엔드포인트 및 S3 경로 확정
- [ ] 동적 라우팅 판단 로직 위치 결정 (게이트웨이 / 서비스 / CDN)
- [ ] Kill Switch 구현 레벨 결정 (WAF / 라우터 / Feature Flag)
- [ ] CDN 캐시 무효화 전략 (전역/부분, TTL, API 연동)
- [ ] 외부 ↔ 내부 통신 시 보안 규칙 확인
- [ ] 보안 정책 및 시큐어 코딩 가이드 정리
- [ ] Audit Log에 필요한 필드 정의 (DevOps 모니터링/장애 분석 관점)
- [ ] Mock API / Swagger 스펙 초안 작성 (읽기/쓰기, 인증 방식)
- [ ] 롤백 전략 확정 (MVP: On/Off만 vs 1단계 롤백 포함)
